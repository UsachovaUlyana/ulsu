# Online Store Transactions

Это проект-решение практического задания "Online Store Transactions". Сервис написан на Python с использованием ORM **SQLAlchemy** и базы данных **PostgreSQL**. Всё приложение запускается изолированно через Docker Compose.

---

## 📂 Структура проекта

* `requirements.txt` — содержит список необходимых Python-библиотек (SQLAlchemy, psycopg2-binary).
* `Dockerfile` — инструкции для сборки легковесного образа приложения на основе `python:3.11-slim`. Устанавливает зависимости и запускает основной скрипт.
* `docker-compose.yml` — конфигурация для запуска 2х сервисов: самой базы данных (`postgres:15`) и нашего приложения (`app`). Настроена зависимость, чтобы приложение запускалось только после того, как БД будет готова принимать подключения (через `healthcheck`).
* `main.py` — главный файл приложения. Содержит описание моделей БД и саму логику всех 3-х сценариев с использованием транзакций.
* `задание.md` — исходный файл с описанием задачи.

---

## 🛠 Выбранный стек технологий

* **Язык**: Python 3.11
* **БД**: PostgreSQL 15
* **Библиотеки**: 
  * `SQLAlchemy 2.0` (современный релиз, используется для декларативного описания таблиц и управления сессиями/транзакциями прямо из кода).
  * `psycopg2-binary` (внутренний драйвер, позволяющий SQLAlchemy общаться с PostgreSQL).

## 🚀 Как запустить

Вам не нужно устанавливать Python или Postgres на свой компьютер — достаточно иметь установленный **Docker** и **Docker Compose**.

1. Откройте терминал в папке проекта (там, где лежит `docker-compose.yml`).
2. Выполните команду:
   ```bash
   docker-compose up --build
   ```
3. Docker скачает образ PostgreSQL, соберет образ с Python, запустит БД, дождется ее готовности, и затем запустит файл `main.py`. Вы увидите вывод в консоль о ходе выполнения транзакций.
   Если захотите остановить приложение и удалить базу, выполните: `docker-compose down -v`.

## ⚙️ Как работают сценарии и атомарность операций

В коде для обеспечения **атомарности** используется конструкция менеджера контекста SQLAlchemy:

```python
with session.begin():
    # ... тут выполняются SQL операции
```
Этот блок автоматически открывает транзакцию (`BEGIN`).
* Если весь код внутри блока выполнится без ошибок, транзакция фиксируется (`COMMIT`).
* Если возникнет ошибка (например, исключение), выполняется откат транзакции (`ROLLBACK`), и никакие частичные данные не сохраняются. Это гарантирует, что база данных всегда остается в консистентном (последовательном) состоянии.

### Сценарий 1: Размещение заказа
Функция `checkout_order`. 
1. В рамках одной транзакции сначала создается пустой `Order`. Делается `.flush()`, чтобы получить его сгенерированный ID.
2. Затем ищутся требуемые продукты. Для каждого продукта рассчитывается сумма (`Price * Quantity`), и соответствующие записи добавляются в таблицу `OrderItems`.
3. Параллельно накапливается `total_amount`, который в конце записывается в `new_order.TotalAmount`.
4. Блок транзакции закрывается, сохраняя заказ, позиции и сумму атомарно. Если хоть на одном шаге что-то пойдет не так (например, ID продукта не будет найден), *весь* заказ отменится.

### Сценарий 2: Обновление email клиента
Функция `update_customer_email`.
1. Атомарно находим клиента по его `CustomerID`.
2. Меняем ему значение `Email` на новое. 
3. Если клиент не найден — выдается ошибка, происходит откат. Если найден — `COMMIT` сохраняет правки.

### Сценарий 3: Добавление нового продукта
Функция `add_new_product`.
1. В рамках транзакции создается новая сущность `Product(ProductName, Price)`.
2. Запись добавляется в сессию. При выходе из блока `with session.begin()` происходит атомарный `COMMIT`. Если бы мы, например, добавляли 10 продуктов разом, и на 9-м возникла ошибка, в БД не попал бы ни один из них.
