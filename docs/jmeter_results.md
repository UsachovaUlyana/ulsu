# Результаты нагрузочного тестирования (JMeter)

**Дата прогона:** 16.05.2026  
**Цель:** оценить производительность `ranking-service` (горячий путь — `GET /feed`) под нагрузкой  
**Тестовый стенд:** macOS + Docker Compose (localhost)

---

## 1. Конфигурация теста

### Файлы

| Файл | Назначение |
|:-----|:-----------|
| `infrastructure/jmeter/dating_load_test.jmx` | JMeter test-plan |
| `infrastructure/jmeter/seed_users.py` | Скрипт посева тестовых пользователей |
| `infrastructure/jmeter/report.jtl` | Сырые результаты (CSV) |
| `infrastructure/jmeter/report/index.html` | HTML-дашборд с графиками |

### Параметры test-plan

| Параметр | Значение | Описание |
|:---------|:---------|:---------|
| `users` | 10 | Виртуальных пользователей (threads) |
| `duration` | 60 с | Длительность теста |
| `ramp` | 10 с | Время разгона до пиковой нагрузки |
| `ranking_port` | 8002 | Порт ranking-service |
| **Think Time** | 100–400 мс | Gaussian Random Timer между запросами |

### Распределение запросов

- **70%** — `GET /api/v1/feed/{telegram_id}` (основной сценарий)
- **30%** — `GET /api/v1/ratings/{telegram_id}` (вспомогательный)

### Предварительная подготовка

Перед прогоном в БД были посеяны **200 тестовых пользователей** через Profile Service API:

```bash
python3 infrastructure/jmeter/seed_users.py
```

Скрипт создал:
- `users` — 200 записей (`telegram_id` из диапазона 100M–999M)
- `profiles` — 200 анкет (имя, возраст 18–35, пол, город, интересы)
- `preferences` — 200 наборов фильтров (пол, возраст, город поиска)

**Зачем:** JMeter использует случайные `telegram_id`. Без реальных пользователей `/feed` возвращает `{"profile": null}` (200 ОК, но без данных), а `/ratings` возвращает 404.

## 2. Исправления test-plan

Исходный `.jmx` не запускался на JMeter 5.6.3 из-за несовместимости формата. Были внесены правки:

| Проблема | Исправление | Где в файле |
|:---------|:------------|:------------|
| `${USERS}` в `<intProp>` | Заменено на `${__P(users,50)}` в `<stringProp>` | ThreadGroup |
| `${RAMP}` в `<intProp>` | Заменено на `${__P(ramp,30)}` в `<stringProp>` | ThreadGroup |
| `${DURATION}` в `<intProp>` | Заменено на `${__P(duration,120)}` в `<stringProp>` | ThreadGroup |
| `${RANKING_PORT}` в `<intProp>` | Заменено на `${__P(ranking_port,8002)}` в `<stringProp>` | HTTP Sampler |
| `<stringProp>200</stringProp>` без `name` | Добавлен `name=""` | ResponseAssertion |
| Отсутствие пауз между запросами | Добавлен **Gaussian Random Timer** (100 мс base + 300 мс range) | ThreadGroup → hashTree |

**Команда запуска:**

```bash
cd infrastructure/jmeter
jmeter -n \
  -t dating_load_test.jmx \
  -l report.jtl \
  -e -o report \
  -Jhost=localhost \
  -Jranking_port=8002 \
  -Jusers=10 \
  -Jduration=60 \
  -Jramp=10
```

### 3.4. Анализ ошибок 404

404 возникает **только** на `/ratings` для `telegram_id`, которых нет в БД. Это ожидаемое поведение:

- `/feed` — возвращает 200 даже для несуществующего пользователя (`{"profile": null}`), потому что логика допускает «пустой» ответ.
- `/ratings` — возвращает 404, если `telegram_id` не найден в таблице `users`.

Поскольку в БД посеяно 200 пользователей из 900M возможных `telegram_id`, вероятность попадания случайного ID ~0.00002%. Поэтому почти все запросы к `/ratings` получают 404.

> **Вывод:** ошибки 404 — это **не проблема производительности**, а следствие тестовой методологии (случайные ID). Они не влияют на оценку latency `/feed`.

## 6. Выводы

1. **Ranking Service выдерживает нагрузку** — p95 latency 9 мс при 35 RPS на macOS + Docker.
2. **BindException полностью устранён** после добавления think time (100–400 мс).
3. **Приложение не возвращает 5xx** — все ошибки либо ожидаемые 404 (несуществующие ID), либо решены на уровне тестовой конфигурации.
4. **Тест-план исправлен** и готов к повторным прогонам.
5. **Для более чистых результатов** рекомендуется:
   - Увеличить количество посеянных пользователей до 1000+ (чтобы `/ratings` чаще попадал в существующие ID).
   - Запускать JMeter из Docker-контейнера в той же сети, что и сервисы (избежать macOS TCP-лимиты).
   - Использовать отдельный стенд (Linux/VPS) для промышленного нагрузочного тестирования.
